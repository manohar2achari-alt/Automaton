Create Automation Account

We need a secure container to execute our PowerShell logic. This resource handles the authentication and scheduling.

  1. Log in to the Azure Portal and search for Automation Accounts.
  2. Click Create.
  3. Basics: Select your Subscription and Resource Group.
  4. Identity Tab (Crucial): Ensure System Assigned Managed Identity is set to On. This creates an identity in Entra ID representing this automation account.
  5. Networking: Select Public Access (unless your organization requires Private Link).
  6. Review and Create.

Import Modules (Runtime 7.2)

By default, Azure Automation does not have the Microsoft Graph SDK installed. We must import it manually to interact with Entra ID.

‚ö†Ô∏è Critical Requirement: You must use Runtime version 7.2 (or higher). The older PowerShell 5.1 runtime does not support the modern Graph commands effectively.
  • Open your new Automation Account.
  • On the left blade, go to Shared Resources > Modules.
  • Click + Add a module.
  • Select Browse from Gallery.
  • Search for and import these two modules (in this order):
    1. Microsoft.Graph.Authentication (Runtime: 7.2)
    2. Microsoft.Graph.Applications (Runtime: 7.2)

Grant Graph Permissions

We need to give our Managed Identity the "Keys to the Kingdom"‚Äîbut only specifically for reading apps and sending mail. This step cannot be done in the Portal GUI; it requires a one-time PowerShell script run by an Admin.

Setup Script (Run on Local Machine) 
# 1. Login as Global Admin
Connect-MgGraph -Scopes "AppRoleAssignment.ReadWrite.All","Application.Read.All"

# 2. CONFIG: Enter your Automation Account Name
$AutoAccountName = "aa-secret-monitor-prod" 

# 3. Get the Service Principals
$MI = Get-MgServicePrincipal -Filter "displayName eq '$AutoAccountName'"
$Graph = Get-MgServicePrincipal -Filter "appId eq '00000003-0000-0000-c000-000000000000'"

# 4. Define Permissions (Read Apps & Send Mail)
$Roles = @("Application.Read.All", "Mail.Send")

# 5. Loop and Assign
foreach ($RoleName in $Roles) {
    $Role = $Graph.AppRoles | Where-Object {$_.Value -eq $RoleName}
    
    New-MgServicePrincipalAppRoleAssignedTo `
        -ServicePrincipalId $MI.Id `
        -PrincipalId $MI.Id `
        -ResourceId $Graph.Id `
        -AppRoleId $Role.Id
        
    Write-Host "SUCCESS: Assigned $RoleName" -ForegroundColor Green
}

The Runbook Logic

This is the engine. It scans your tenant and triggers one email per expiring secret. This ensures your ticketing system (e.g., ServiceNow, Jira) creates individual tickets for each issue.

  • Go to Process Automation > Runbooks > Create a runbook.
  • Name: Monitor-AppSecrets
  • Runbook Type: PowerShell
  • Runtime Version: 7.2
  • Paste the code below, Save, and Publish.
Runbook Source Code (PowerShell 7.2) 
# --- CONFIGURATION ---
$NotifyDays     = @(60, 30, 7)
$SenderEmail    = "alerts@yourdomain.com"  # Must match a real mailbox
$RecipientEmail = "iam-team@yourdomain.com"

try {
    # --- AUTHENTICATION ---
    # We use the Managed Identity to get a token for Graph
    Connect-AzAccount -Identity
    $token = Get-AzAccessToken -ResourceUrl "https://graph.microsoft.com"
    Connect-MgGraph -AccessToken $token.Token

    $Today = Get-Date
    Write-Output "Starting scan of App Registrations..."
    
    # Fetch all apps with their credential details
    $Apps = Get-MgApplication -All -Property DisplayName, AppId, PasswordCredentials

    foreach ($App in $Apps) {
        foreach ($Cred in $App.PasswordCredentials) {
            $Expiry = Get-Date $Cred.EndDateTime
            $DaysLeft = ($Expiry - $Today).Days

            # Check if the days remaining matches our alert window
            if ($NotifyDays -contains $DaysLeft) {
                
                $AppName = $App.DisplayName
                $AppId   = $App.AppId
                $ExpDate = $Expiry.ToString("yyyy-MM-dd")
                
                # Construct HTML Email
                $HtmlBody = "
                <div style='font-family: sans-serif; color: #333;'>
                    <h3 style='color: #d9534f;'>‚ö†Ô∏è Action Required: Secret Expiring</h3>
                    <p>The secret for the following application is expiring in <strong>$DaysLeft days</strong>.</p>
                    <div style='background: #f9f9f9; padding: 15px; border-left: 4px solid #d9534f;'>
                        <p><strong>App Name:</strong> $AppName</p>
                        <p><strong>App ID:</strong> $AppId</p>
                        <p><strong>Expiry Date:</strong> $ExpDate</p>
                    </div>
                    <p>Please rotate this secret immediately to prevent service interruption.</p>
                </div>"

                $Params = @{
                    Message = @{
                        Subject = "ALERT: Secret for '$AppName' expires in $DaysLeft days"
                        Body = @{ ContentType = "HTML"; Content = $HtmlBody }
                        ToRecipients = @( @{ EmailAddress = @{ Address = $RecipientEmail } } )
                    }
                    SaveToSentItems = "false"
                }
                
                Send-MgUserMail -UserId $SenderEmail -BodyParameter $Params
                Write-Output " [ALERT SENT] $AppName ($DaysLeft days remaining)"
            }
        }
    }
} catch {
    Write-Error "Runbook failed with error: $_"
}

Testing & Scheduling

Never assume it works until you see the email in your inbox.

Testing

  • Inside the Runbook, click Test pane (top menu).
  • Click Start.
  • Watch the logs. If you have an app expiring in 60, 30, or 7 days, you should see [ALERT SENT] in the output and receive an email.

Scheduling

  • Go back to the Automation Account blade.
  • Click Schedules > + Add a schedule.
  • Set it to start tomorrow at 08:00 AM and recur Every 1 Day.
  • Go to your Runbook > Link to schedule and select the schedule you just created.
